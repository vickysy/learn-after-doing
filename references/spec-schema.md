# 拆解规格 SPEC —— 字段说明

页面从这份 `SPEC` 对象自动生成。字段越准、类比越接地气，小白越看得懂。

## 顶层字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `title` | string | 一句话标题，句式"X 是怎么做到 Y 的"。例：`剪藏插件：一个网页是怎么进你的 Get 笔记的` |
| `essence` | string(HTML) | 一句话本质，**必须用生活类比**。可用 `<b>` 加粗关键词。 |
| `nodes` | array | 信息流上的节点，3~5 个，从起点到终点排序 |
| `flows` | array | 节点之间的流动，长度 = `nodes.length - 1`，按顺序 |
| `concepts` | array | 关键技术概念，3~5 个，点开渐进展开 |
| `watch` | string[] | 小白只需盯的点（3 个最佳），可用 `<b>` |
| `failures` | array | 常见故障 → 原因 → 怎么修 |
| `pattern` | string | 可迁移的本质模式，一句话（举一反三的钥匙） |
| `similar` | array | 3~6 个同模式的相似场景，激发举一反三 |
| `inspiration` | array | 联网搜到的真实相似案例（**只放真链接**） |

## nodes[]

| 字段 | 说明 |
|------|------|
| `id` | 唯一标识（英文短词，如 `page`/`ext`/`server`/`app`），供 `flows` 引用 |
| `icon` | 一个 emoji，直观代表这个节点（📄🧩☁️📱🗄️🔌🧠 等） |
| `name` | 节点名，≤6 字 |
| `tech` | 这一步用到的技术，短标签（如 `API 接口 + 数据库`） |
| `plain` | 小白版一句话：这一步到底在干嘛 |

## flows[]

| 字段 | 说明 |
|------|------|
| `from` / `to` | 两端节点的 `id` |
| `label` | 这段流动在做什么，短（如 `读取+打包`、`API 请求 · 带令牌`）。用 ` · ` 分隔会在箭头处换行 |
| `detail` | 小白版详细解释，播放到这一步时显示在字幕区 |
| `key` | 布尔。标 `true` 的是**最关键一环**（通常是鉴权/发请求），页面高亮成黄色 |

## concepts[]

| 字段 | 说明 |
|------|------|
| `term` | 概念名（如 `令牌 token`） |
| `plain` | 一句话人话解释（折叠时也显示） |
| `more` | 再深一层，但仍是人话，点开后显示 |

## failures[]

| 字段 | 说明 |
|------|------|
| `symptom` | 用户会看到的现象（如 `点了没反应`） |
| `cause` | 为什么会这样（人话） |
| `fix` | 怎么自查 / 修 |

## 设计原则
- **信息流优先**：`nodes`/`flows` 讲的是"数据怎么走"，不是功能清单。
- **无生词**：每个技术词配类比。令牌=工牌、服务器=仓库、API=打电话、数据库=仓库货架、插件=跑腿小哥。
- **故障映射盯点**：`failures` 里最容易坏的地方，应该就是 `watch` 里让人盯的地方。
- **克制深度**：目标是"看懂+会修"，不是教开发。每个 `more` 一两句即可。
