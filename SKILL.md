# 拆盲盒（反向科普）

把一个「AI 帮我做好了但我不懂原理」的黑箱，拆成一个小白能看懂、能互动的可视化网页。

**核心信念**：小白不用成为开发者，但要知道背后的原理和"坏了怎么修"，否则永远停在表面，下次还是不会。

## 三层架构（换平台只换最外层）

1. **读物证** —— 读完成该任务的工程文件 + 对话上下文（本文件第 1、2 步）
2. **抽规格** —— 从物证里提炼一份结构化「拆解规格 SPEC」（第 3 步，纯数据，跟平台无关）
3. **渲染** —— 把 SPEC 灌进 `assets/template.html` 生成可交互网页（第 4 步）

> 移植到 Codex / WorkBuddy 等：第 1、3 步换成对应平台的文件读取与渲染方式，第 2 步的 SPEC 格式不变。

---

## 第 1 步：确定模式，定位相关文件

判断是哪种情况：

- **A. 任务刚在本对话里做完** —— 你（AI）清楚自己刚创建 / 改动了哪些文件。直接把这些文件作为物证。
- **B. 回头拆解一个已有产物** —— 让用户指一个**文件夹或入口**（"拆解我这个剪藏插件，代码在 xxx"）。若用户只给了一句话没给文件，先问清楚产物在哪，或让他描述这是个什么东西。

**必须真正读文件，不要凭记忆猜。** 代码不会撒谎。

## 第 2 步：读工程文件（找关键"物证"）

按产物类型，重点读这些（读不到就跳过，不要编造）：

- **清单 / 配置**：`manifest.json`、`package.json`、`*.config.*`、`.env.example` —— 看它是什么、依赖了什么、要什么权限
- **入口 / 主逻辑**：主脚本、`content script` / `background`、`main.*`、路由文件 —— 看它启动后干了什么
- **对外通信**：发网络请求的地方（`fetch` / `axios` / `requests` / API 调用）、连数据库 / 存储的地方、鉴权 / token 的地方 —— **这些就是信息流的关键节点**
- **README / 注释** —— 作者自己怎么描述这个东西

一边读，一边在心里回答：**用户一操作，数据从哪儿出发 → 经过谁 → 用什么方式传 → 最后存/显示到哪儿？中间哪几步最容易坏？**

## 第 3 步：抽出「拆解规格 SPEC」

把读到的东西填成下面这个对象（字段说明见 `references/spec-schema.md`，完整实例见 `assets/template.html` 里的默认 SPEC）。

```js
const SPEC = {
  title: "一句话标题：xxx 是怎么做到 yyy 的",
  essence: "一句话本质，用<b>生活类比</b>说清它到底是个啥（可用 <b> 加粗关键词）",
  nodes: [   // 信息流上的 3~5 个节点，从起点到终点
    {id:"唯一id", icon:"📄", name:"节点名(≤6字)", tech:"用到的技术(短)",
     where:"你的电脑"/"云端"/"浏览器里"(可选,在哪运行), scope:"local"/"cloud"(可选,配 where 上色),
     plain:"小白版一句话：这一步在干嘛"}
  ],
  flows: [   // 节点之间的流动，数量 = nodes-1，按顺序
    {from:"起点id", to:"终点id", label:"这段在做什么(短)",
     net:"local"/"online"(可选,这一跳出不出网), detail:"这一步的小白版详细解释", key:true/false}
  ],
  concepts: [   // 3~5 个关键技术概念，点开渐进展开
    {term:"概念名", plain:"一句话人话解释", more:"再深一层，但仍是人话"}
  ],
  watch: [   // 小白只需盯的 3 个点（链路上最容易断的环），可用 <b>
    "盯点一"
  ],
  failures: [   // 常见故障 → 大白话原因 → 怎么办 → 一句发给 AI
    {symptom:"用户会看到的现象(大白话)", cause:"大白话原因，能类比就类比",
     fix:"小白能做的动作。主语是「你回去跟 AI 说 / 你在哪点一下」，不是「你去改哪个文件」",
     sayToAI:"(可选)一句能直接复制、丢回给 AI 让它修的话，可含具体文件/变量名——这句是给 AI 看的"}
  ],
  pattern: "一句话点破可迁移的本质模式（举一反三的钥匙）。例：浏览器抓当前页→带身份→存进某个云端账号",
  similar: [   // 3~6 个「种类不同」的玩法，不是同一件事换对象
    {icon:"👀", title:"场景名(短)", reuses:"复用了哪块零件(短)", what:"一句话它是啥、跟原来差在哪",
     sayToAI:"一句小白口吻的具体场景「我经常…想要…」，页面渲染成一键按钮"}
  ],
  method: "教小白自己举一反三的一句话思路（可用 <b>）",
  inspiration: [   // 联网搜到的真实相似案例——只放真链接，禁止编造
    {title:"标题", source:"Reddit / GitHub / 博客", url:"https://真实链接"}
  ]
};
```

### 写作铁律（面向小白）
- **每个技术词都要配一句人话或类比**（令牌=工牌、服务器=仓库、API=打电话、插件=跑腿小哥）。宁可类比不精确，也不要留生词。
- `nodes` 聚焦"信息怎么流"，不是罗列功能。3~5 个，别更多。
- `flows[].key:true` 标在**最关键的一环**（通常是鉴权/发请求那步），页面会高亮它。
- `failures` 必须映射到 `watch` 的几个点——最容易坏的地方就是最该盯的地方。
- 语气像木妈：亲切、干脆、"这其实就是……的事"。
- **联网边界必须说清（小白最容易误会）**：给每个 `node` 标 `where`（你的电脑 / 云端 / 浏览器里）、每个 `flow` 标 `net`（`local` 本机内 / `online` 出网）。**"本地服务"≠"不联网"**——"只监听本机"说的是别人连不进来，它自己照样能主动出网。凡是最终写到云端（Get笔记、飞书、任何在线账号）的产物，一定有某一跳是 `online`，别把整条链说成"全程本地/不联网"。
- **别默认"有服务/有 AI 工具就要大模型"**：很多中转/上传/读取的程序只是"传声筒"，不思考不生成、不用任何模型。该说明白的就说"它不是 AI，只负责转交/读取"。只有真用到大模型的节点才提模型。
- **「坏了怎么修」= 让小白理解 + 一键交给 AI 修**（这是最容易翻车的地方）：
  - 用户是「借 AI 做出这东西的小白」，他的修理工具就是 AI。**默认动作不是"自己去弄"，而是"一键把问题甩给 AI"**。
  - **每一条故障都必须给 `sayToAI`**（不是可选！）：一句用户能一键复制、丢回给 AI 就能修的话。页面把它渲染成主操作——带「复制发给 AI」按钮的蓝卡。`sayToAI` 要**自带上下文**（点名是什么产物 + 现象），这样 AI 拿到就能直接上手；里面可以带文件名/变量名/命令（那是给 AI 读的）。
  - `cause`（大白话原因）负责让小白**看懂为什么**——这是"知其所以然"，尽量配生活类比（"还开着演示模式=样板间的家具，看着有其实不能用"）。**技术标识符不许进 `symptom`/`cause` 的主句**。
  - `fix` 只是给愿意自己动手的人看的**次要**一行（页面显示成灰字「想自己弄：…」），可留空。绝大多数小白只会点那个「交给 AI」。
  - 自检：把 `cause` 念给你妈听她懂不懂"哪儿坏了"；再看她是不是只要点一下「复制发给 AI」就行。

### 举一反三（务必做，这是学习闭环的最后一口气）

小白复盘完原理后，要让他冒出一句"哦，这套还能拿去干别的！"。填三块：

1. **`pattern`** —— 从这次的拆解里**抽出可迁移的本质模式**（不是这一个产物，是背后那类做法）。这是举一反三的钥匙，一句话点破。
2. **`similar`** —— 3~6 个玩法，但**必须"种类不同"，不是同一件事换个对象**（这是最容易做砸的地方）。
   - ❌ 反面：存到育儿库 / 存到 AI 库 / 存到菜谱库……全是"剪藏到别处"，换汤不换药，小白看了不会"哇"。
   - ✅ 正面：先把这东西**拆成几块零件**（如：抓网页 / 本机服务遥控器 / 存云端 / CLI 一条龙），然后让每张卡**复用不同的零件、干出不一样种类的事**——盯梢（抓网页那块）、遥控电脑（本机服务那块）、加个 AI 判断（换中间环节）、抓→译→存→提醒一条龙（串工具那块）。
   - 每张卡填 `reuses`：**复用了哪块零件**（页面会标成小标签），让小白看见"原来它是可拆可拼的乐高"。
   - `sayToAI` 用小白口吻写具体场景「我经常…想要…」，**绝不许出现机制术语**（本机服务/凭证/骨架/API…）。AI 自己会把原理套上去。
   - 场景贴近这个用户的真实身份/受众，别都是程序员味儿的。
3. **`method`** —— 一句话教小白**自己**举一反三的思路（如"拆完问自己：它由哪几块拼的？每块还能干嘛？"）。授人以渔，别只给答案。
4. **`inspiration`** —— **用联网搜索**（WebSearch / 上 Reddit、GitHub、技术社区）找**真实的**相似项目/讨论，挑 2~4 条放进来，带真链接。
   - **只放你真搜到的链接，一条都不许编造**。搜不到宁可留空，也不造假 URL。
   - 搜的是"别人拿这套模式还做了什么"，给用户真实的兔子洞去逛。

## 第 4 步：生成可交互网页

1. 读 `assets/template.html`。
2. 把 `/*__SPEC_START__*/` 与 `/*__SPEC_END__*/` 之间的内容，整体替换成你在第 3 步写好的 `const SPEC = {…};`。**其余部分（渲染器、样式）一律不动。**
3. 写出到用户工作目录，文件名建议 `拆盲盒_<产物名>.html`（放桌面或当前项目目录）。
4. 用系统默认浏览器打开给用户看（`open <文件路径>`）。页面加载后会自动播一遍信息流，用户可点节点、点开概念、展开故障。

## 交付话术
生成后，用一两句告诉用户：这个页面能**点**——点节点看那一步在干嘛、点概念展开讲解、点故障看怎么修；并且它本身就是一条现成的**小红书/公众号素材**（回答"这到底怎么做出来的"）。

## 边界
- 只做**解释性可视化**，不改用户的工程、不重写代码。
- 读不到的细节**如实说不知道**，不要编造技术栈或接口。
- 目标是"看懂 + 会修"，不是培养开发者；深度点到为止。
